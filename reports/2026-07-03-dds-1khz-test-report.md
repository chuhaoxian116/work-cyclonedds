# Cyclone DDS 1 kHz 通信测试报告

- 测试日期：2026-07-03，追加测试：2026-07-06
- Cyclone DDS：0.10.5
- 消息频率：1 kHz 基准；2 kHz、5 kHz 频率压力；1 kHz CPU 压力
- 消息大小：约 128 B 基准；32～64 KiB 负载测试
- 网络传输：UDP
- DDS Domain：0
- 实时配置候选：CPU `1,2`、`SCHED_FIFO 40`

## 1. 测试目标

本次测试分为两条互补的验证路径：

1. 使用 `ddsperf ping/pong` 测量往返通信能力。ddsperf 报告的是
   `RTT / 2`，适合比较绑核和实时优先级对通信时延的影响。
2. 使用 C++ `arm_state_1khz` Demo 模拟机械臂状态流。Publisher 每
   1 ms 发布一次，Subscriber 检查应用可见接收周期、抖动和序号连续性。

两组数据的口径不同，不能把 ddsperf 的 `RTT / 2` 与 C++ 的相邻接收
间隔直接当作同一种“网络延迟”比较。C++ Demo 未使用 PTP 统一两台机器
的时钟，因此没有计算跨主机单向传输时延。

## 2. 测试环境

发送端已确认使用 PREEMPT_RT 内核：

```text
Linux 5.15.148-rt74 PREEMPT_RT
4 个物理核 / 8 个逻辑 CPU
CPU 0,4 -> core 0
CPU 1,5 -> core 1
CPU 2,6 -> core 2
CPU 3,7 -> core 3
```

网卡 `enp0s31f6` 的 IRQ 主要位于 CPU 7。因此选择 CPU `1,2`，既避开
网卡 IRQ，又使用两个不同的物理核。内核实时调度限额为：

```text
sched_rt_period_us  = 1000000
sched_rt_runtime_us = 950000
```

## 3. ddsperf 测试过程

### 3.1 原始基准

最初使用普通调度、不绑核的方式运行：

```text
ddsperf ping 1kHz size 128 listener
```

原计划运行 28800 秒，但 SSH 会话退出后进程收到 `SIGHUP`，实际运行
5211 秒。因此后续统一使用 5211 秒作为长时间测试基准，并改用
`sudo setsid -f ...` 脱离终端。

### 3.2 单逻辑核 + FIFO 40

Ping 和 Pong 都限制在 CPU 2，并使用 `SCHED_FIFO 40`。典型时延和
超时率改善，但 ddsperf 的工作线程和每秒统计输出共用一个逻辑核，
几乎每秒都出现一次约 1.25～1.5 ms 的峰值，最坏值反而变差。

### 3.3 双物理核 + FIFO 40

将进程允许在 CPU `1,2` 上运行，优先级保持 `SCHED_FIFO 40`，先进行
600 秒筛选。结果明显优于前两种配置，因此被选为后续推荐配置。

## 4. C++ 1 kHz Demo

### 4.1 通信模型与 QoS

Demo 位于 `myproject/samples/arm_state_1khz`，采用单向最新状态流：

```text
Publisher -- 1 ms/帧 --> Subscriber
```

使用的 QoS 为：

```text
Reliability   BestEffort
Durability    Volatile
History       KeepLast(1)
Deadline      2 ms
Lifespan      5 ms
```

- `BestEffort + KeepLast(1)` 优先保证新鲜状态，不会让旧状态在队列中堆积。
  它不承诺每帧必达，所以消息中加入 `sequence` 检查缺帧。
- `Deadline=2 ms` 是周期健康监视器：连续约两个发送周期没有新数据时报告
  Deadline miss。它不会等待或重发数据。
- `Lifespan=5 ms` 是样本年龄上限：超过五个控制周期的旧状态不再有消费
  价值。5 ms 是本次测试起点，产品中应根据机械臂允许的最大状态年龄调整。
- `Volatile` 表示只接收在线期间发布的数据，不向后来加入的 Reader 补历史。

### 4.2 Demo 优化过程

| 轮次 | 主要变化 | Publisher 结果 | Subscriber 结果 | 结论 |
|---|---|---|---|---|
| 1 | 每 60 s 排序并打印 | 写 599,652；跳 348；最大 37.185 ms | 收 599,533；缺口 465；最大 37.237 ms | 实时线程内排序/输出造成停顿 |
| 2 | 移除周期日志 | 写 599,970；跳 30；最大 19.032 ms | 收 599,948；缺口 39；最大 19.172 ms | 明显改善，但仍有动态扩容 |
| 3 | 增加诊断并预分配 | 远端仍运行旧 Publisher | 缺口出现在 65,539、131,079、262,158、524,320 附近 | `vector` 倍增扩容特征，同时发现部署版本不一致 |
| 4 | 两端均部署最终版本 | 写 600,000；跳 0；Deadline miss 0 | 收 599,992；缺口 0；最大间隔 1.301 ms | 600 秒筛选通过 |
| 5 | 最终版本长测 | 写 5,211,000；跳 0；Deadline miss 0 | 有效序列连续；缺口 0；最大间隔 1.377 ms | 5211 秒长测通过 |

最终版本采用以下措施：

- 使用绝对时间 `sleep_until()`，避免 `sleep_for()` 将每周期误差累计成漂移。
- 唤醒过晚时跳过已经失效的周期，不追赶式连发，并在序号中保留缺口。
- 所有统计数组在计时前按预计样本数 `reserve()`。
- 1 kHz 单线程数据路径中不排序、不打印，只在结束时输出汇总。
- 分别记录原始唤醒延迟、`write()` 耗时、统计耗时和 `take()` 耗时。

### 4.4 C++ 后续对比的统一指标口径

后续 C++ 基准、频率压力和 CPU 压力测试均使用同一组核心指标对比。这样
避免把不同语义的数字混在一起：

| 指标 | 所属端 | 含义 | 通过时重点 |
|---|---|---|---|
| Publisher 写入 / 跳周期 | Publisher | 是否按计划完成发送循环 | 写入数符合预期，跳周期为 0 |
| Publisher 间隔 P99 / 最大 | Publisher | 本机发送循环相邻两次 `write()` 的周期 | 最大值不要明显越过业务周期预算 |
| Wake P99 / 最大 | Publisher | 发送线程相对计划唤醒时刻的延迟 | 判断本机调度抖动 |
| Write 平均 / P99 / 最大 | Publisher | DDS `write()` 调用耗时 | 判断写入是否阻塞或异常变慢 |
| Subscriber 接收数 | Subscriber | 应用实际取到的样本数 | 结合首个序号和运行窗口解释 |
| Sequence gaps | Subscriber | 应用层序号缺口 | 判断是否有状态被覆盖或丢失 |
| Subscriber 间隔 P99 / 最大 | Subscriber | 应用可见的相邻接收周期 | 判断接收侧尾延迟 |
| Take 平均 / 最大 | Subscriber | DDS `take()` 调用耗时 | 判断读取路径是否异常变慢 |
| Deadline / lost / rejected | Subscriber | DDS 状态计数 | 只能辅助判断，不能替代 `sequence` |

其中最适合横向对比的是：

```text
Publisher 跳周期
Subscriber sequence_gaps
Publisher 最大间隔 - 目标周期
Subscriber 最大间隔 - 目标周期
Wake 最大值
Write 最大值
Take 最大值
```

对于 1 kHz、2 kHz、5 kHz 这类不同频率测试，直接比较“最大间隔”会受目标
周期影响。因此图表中优先使用“最大超周期量”：

```text
最大超周期量 = 最大相邻周期 - 目标周期
```

例如 1 kHz 目标周期是 1000 µs，Subscriber 最大间隔为 1377.104 µs，
则最大超周期量为 377.104 µs。这个值比直接比较 1377.104 µs 更适合跨频率
观察调度和接收尾部抖动。

## 5. 结论

1. 当前机器上，CPU `1,2` + `SCHED_FIFO 40` 是今天筛选出的最佳配置。
   单独绑定一个逻辑核会让 ddsperf 内部线程互相争用，不建议使用。
2. C++ 最终版本连续运行 5211 秒，Publisher 没有错过任何 1 ms 周期；
   Subscriber 在加入通信后收到完整连续序列。
3. 当前结果证明的是“本测试环境下的稳定 1 kHz 最新状态传输”，不是
   DDS BestEffort 的形式化必达保证。产品逻辑仍应监视 sequence、Deadline
   和状态年龄，并为通信中断设计安全降级。
4. 如果要测真实单向网络时延，需要两端使用 PTP/硬件时间戳同步，并在消息
   中携带同一时钟域的发送时间。
5. 2026-07-06 的 CPU 压力结果进一步表明：Publisher 在所有 CPU 档位中
   都保持零跳周期；但 BestEffort + KeepLast(1) 在部分接收压力场景会覆盖
   中间状态，不能用于证明“每一帧必达”。完整结果见 7.8。

## 6. 复测建议

- 短筛选：先运行 600 秒，确认没有序号缺口、跳周期和大于 2 ms 的间隔。
- 长测：再运行 5211 秒。正式对比采用 Subscriber 比 Publisher 多运行，
  保证完整覆盖发送窗口；分析时排除 Publisher 停止后的预期 Deadline
  miss，以活跃发送期间的序号连续性判断丢帧。
- 每次保存内核、CPU 拓扑、IRQ、路由、配置和二进制哈希。
- 更新程序后先比较二进制 SHA-256，防止两台机器部署版本不一致。
- 测试期间不要在实时线程中打印逐帧或周期性日志。

可复用脚本位于 `performance/dds_1khz`，使用方法见该目录的
`README.md`。

应用序号必然连续，应以 `sequence_gaps` 为准。

5 kHz 已越过本测试的“零序号缺口”边界，但这不影响正式 1 kHz 结论。
已有结果表明 1 kHz 长测通过、2 kHz 三轮通过，5 kHz 开始出现少量缺口。

#### 7.8.3 CPU 压力结论

1. Publisher 在全部 CPU 档位和 CPU-6 三轮复测中均按计划完成 1 kHz
   发送，说明绝对时间调度、CPU `1,2` 和 FIFO 40 对发送侧保护有效。
2. 接收侧在 CPU-0、CPU-2 和 CPU-6 出现应用层单帧缺口。CPU-6 三轮均
   复现，说明两端同时争用 DDS 物理核时存在稳定但极低概率的覆盖风险。
3. `BestEffort + KeepLast(1)` 的语义是优先保留最新状态，而不是每帧必达。
   如果机械臂业务只消费最新状态，可以把当前结果作为降级概率数据，但仍
   必须监控 sequence、Deadline 和状态年龄。
4. 如果业务要求每一帧都被应用读取，当前 QoS 不满足需求。下一阶段应对比
   `BestEffort + KeepLast(8/16)` 与 `Reliable + KeepLast(8/16)`，同时
   评估排队、追赶式读取、Writer 阻塞和状态陈旧风险。

表中的“最差”同样表示字段级最坏值；如果要追溯某个具体异常，应回到前面
各轮详细表格查看对应 Run。

| 场景 | 目标周期 | 接收数 | Sequence gaps | 间隔 P99 / 最大 | 最大超周期量 | Take 最大 | 结论 |

| 1 kHz 长测基准 | 1000 µs | 5,210,998 | 0 | 1200.257 / 1377.104 µs | 377.104 µs | 137.987 µs | 通过 |
| 1 kHz 冒烟复测 | 1000 µs | 59,998 | 0 | 1198.520 / 1308.254 µs | 308.254 µs | 33.043 µs | 通过 |
| 2 kHz 三轮最差 | 500 µs | — | 0 | 624.011 / 722.938 µs | 222.938 µs | 38.240 µs | 通过 |
| 5 kHz 边界 | 200 µs | 2,999,968 | 9 | 272.167 / 418.924 µs | 218.924 µs | 30.731 µs | 序号未通过 |
| 1 kHz CPU 压力首轮最差 | 1000 µs | — | 2 | 1201.562 / 2102.962 µs | 1102.962 µs | — | 未通过 |
| 1 kHz CPU-6 三轮合计 | 1000 µs | — | 7 / 1,800,000 | 1147.827 / 2089.989 µs | 1089.989 µs | — | 未通过 |

说明：

- `接收数` 在不同主机时钟和不同起止窗口下不能单独用于判断丢包，正式判断
  以 `sequence_gaps` 为准。
- CPU 压力表中的 `1 kHz CPU 压力首轮最差` 使用所有首轮 CPU 档位中的最坏
  Subscriber 最大间隔；`1 kHz CPU-6 三轮合计` 使用 CPU-6 三轮复测中的
  合计缺口和最坏最大间隔。
- `requested_deadline_missed` 在 Subscriber 比 Publisher 多运行的尾段会
  按 `尾段时长 / Deadline` 增加，这部分是预期现象，不能直接解释为活跃
  传输期间丢包。

从这组同口径图可以看出：

1. 1 kHz 长测、1 kHz 冒烟和 2 kHz 三轮都保持 `sequence_gaps=0`。
2. 5 kHz 虽然最大超周期量不高，但出现 9 个序号缺口，说明仅看时间间隔会
   漏掉覆盖事件。
3. CPU 压力下的主要风险是接收侧尾延迟和极少量单帧缺口，而不是发送端跳
   周期。
4. 因此后续 QoS 压力测试应继续把 `sequence_gaps` 放在第一优先级，同时
   对比 `BestEffort + KeepLast(8/16)` 与 `Reliable + KeepLast(8/16)`。
