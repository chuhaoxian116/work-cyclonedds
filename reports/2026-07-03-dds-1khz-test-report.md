# Cyclone DDS 1 kHz 通信测试报告

- 测试日期：2026-07-03，追加测试：2026-07-06
- Cyclone DDS：0.10.5
- 消息频率：1 kHz 基准；2 kHz、5 kHz 压力测试
- 消息大小：约 128 B 基准；32～64 KiB 负载测试
- 网络传输：UDP
- DDS Domain：0
- 实时配置候选：CPU `1,2`、`SCHED_FIFO 40`

## 1. 测试目标

本次测试分为两条互补的验证路径：

1. 使用 `ddsperf ping/pong` 测量往返通信能力。ddsperf 报告的是
   `RTT / 2`，适合比较绑核和实时优先级对通信时延的影响。
2. 使用 C++ `arm_state_1khz` Demo 模拟机械臂状态流。Publisher 每
   1 ms 发布一次，Subscriber 检查应用可见接收周期、抖动和序号连续性。

两组数据的口径不同，不能把 ddsperf 的 `RTT / 2` 与 C++ 的相邻接收
间隔直接当作同一种“网络延迟”比较。C++ Demo 未使用 PTP 统一两台机器
的时钟，因此没有计算跨主机单向传输时延。

## 2. 测试环境

发送端已确认使用 PREEMPT_RT 内核：

```text
Linux 5.15.148-rt74 PREEMPT_RT
4 个物理核 / 8 个逻辑 CPU
CPU 0,4 -> core 0
CPU 1,5 -> core 1
CPU 2,6 -> core 2
CPU 3,7 -> core 3
```

网卡 `enp0s31f6` 的 IRQ 主要位于 CPU 7。因此选择 CPU `1,2`，既避开
网卡 IRQ，又使用两个不同的物理核。内核实时调度限额为：

```text
sched_rt_period_us  = 1000000
sched_rt_runtime_us = 950000
```

## 3. ddsperf 测试过程

### 3.1 原始基准

最初使用普通调度、不绑核的方式运行：

```text
ddsperf ping 1kHz size 128 listener
```

原计划运行 28800 秒，但 SSH 会话退出后进程收到 `SIGHUP`，实际运行
5211 秒。因此后续统一使用 5211 秒作为长时间测试基准，并改用
`sudo setsid -f ...` 脱离终端。

### 3.2 单逻辑核 + FIFO 40

Ping 和 Pong 都限制在 CPU 2，并使用 `SCHED_FIFO 40`。典型时延和
超时率改善，但 ddsperf 的工作线程和每秒统计输出共用一个逻辑核，
几乎每秒都出现一次约 1.25～1.5 ms 的峰值，最坏值反而变差。

### 3.3 双物理核 + FIFO 40

将进程允许在 CPU `1,2` 上运行，优先级保持 `SCHED_FIFO 40`，先进行
600 秒筛选。结果明显优于前两种配置，因此被选为后续推荐配置。

## 4. C++ 1 kHz Demo

### 4.1 通信模型与 QoS

Demo 位于 `myproject/samples/arm_state_1khz`，采用单向最新状态流：

```text
Publisher -- 1 ms/帧 --> Subscriber
```

使用的 QoS 为：

```text
Reliability   BestEffort
Durability    Volatile
History       KeepLast(1)
Deadline      2 ms
Lifespan      5 ms
```

- `BestEffort + KeepLast(1)` 优先保证新鲜状态，不会让旧状态在队列中堆积。
  它不承诺每帧必达，所以消息中加入 `sequence` 检查缺帧。
- `Deadline=2 ms` 是周期健康监视器：连续约两个发送周期没有新数据时报告
  Deadline miss。它不会等待或重发数据。
- `Lifespan=5 ms` 是样本年龄上限：超过五个控制周期的旧状态不再有消费
  价值。5 ms 是本次测试起点，产品中应根据机械臂允许的最大状态年龄调整。
- `Volatile` 表示只接收在线期间发布的数据，不向后来加入的 Reader 补历史。

### 4.2 Demo 优化过程

| 轮次 | 主要变化 | Publisher 结果 | Subscriber 结果 | 结论 |
|---|---|---|---|---|
| 1 | 每 60 s 排序并打印 | 写 599,652；跳 348；最大 37.185 ms | 收 599,533；缺口 465；最大 37.237 ms | 实时线程内排序/输出造成停顿 |
| 2 | 移除周期日志 | 写 599,970；跳 30；最大 19.032 ms | 收 599,948；缺口 39；最大 19.172 ms | 明显改善，但仍有动态扩容 |
| 3 | 增加诊断并预分配 | 远端仍运行旧 Publisher | 缺口出现在 65,539、131,079、262,158、524,320 附近 | `vector` 倍增扩容特征，同时发现部署版本不一致 |
| 4 | 两端均部署最终版本 | 写 600,000；跳 0；Deadline miss 0 | 收 599,992；缺口 0；最大间隔 1.301 ms | 600 秒筛选通过 |
| 5 | 最终版本长测 | 写 5,211,000；跳 0；Deadline miss 0 | 有效序列连续；缺口 0；最大间隔 1.377 ms | 5211 秒长测通过 |

最终版本采用以下措施：

- 使用绝对时间 `sleep_until()`，避免 `sleep_for()` 将每周期误差累计成漂移。
- 唤醒过晚时跳过已经失效的周期，不追赶式连发，并在序号中保留缺口。
- 所有统计数组在计时前按预计样本数 `reserve()`。
- 1 kHz 单线程数据路径中不排序、不打印，只在结束时输出汇总。
- 分别记录原始唤醒延迟、`write()` 耗时、统计耗时和 `take()` 耗时。

## 5. 结论

1. 当前机器上，CPU `1,2` + `SCHED_FIFO 40` 是今天筛选出的最佳配置。
   单独绑定一个逻辑核会让 ddsperf 内部线程互相争用，不建议使用。
2. C++ 最终版本连续运行 5211 秒，Publisher 没有错过任何 1 ms 周期；
   Subscriber 在加入通信后收到完整连续序列。
3. 当前结果证明的是“本测试环境下的稳定 1 kHz 最新状态传输”，不是
   DDS BestEffort 的形式化必达保证。产品逻辑仍应监视 sequence、Deadline
   和状态年龄，并为通信中断设计安全降级。
4. 如果要测真实单向网络时延，需要两端使用 PTP/硬件时间戳同步，并在消息
   中携带同一时钟域的发送时间。

## 6. 复测建议

- 短筛选：先运行 600 秒，确认没有序号缺口、跳周期和大于 2 ms 的间隔。
- 长测：再运行 5211 秒。需要避免停止后的 Deadline miss 时，可让
  Publisher 比 Subscriber 多运行约 2 秒；此时以测量窗口内的序号连续性
  判断丢帧，不直接比较两端总数。
- 每次保存内核、CPU 拓扑、IRQ、路由、配置和二进制哈希。
- 更新程序后先比较二进制 SHA-256，防止两台机器部署版本不一致。
- 测试期间不要在实时线程中打印逐帧或周期性日志。

可复用脚本位于 `performance/dds_1khz`，使用方法见该目录的
`README.md`。

### 7.3 C++ 5 kHz 边界测试

周期改为 200 µs，Subscriber 运行 600 秒，Publisher 运行 602 秒。

Publisher：

```text
scheduled/written       3,010,000 / 3,010,000
skipped_cycles          0
offered_deadline_missed 0
interval P99/max        203.502 / 343.623 us
wake max                146.318 us
write max               123.766 us
```

Subscriber：

```text
received                2,999,968
sequence_gaps           9
duplicates/out_of_order 0 / 0
interval P99/max        272.167 / 418.924 us
take max                30.731 us
deadline/lost/rejected  0 / 0 / 0
```

9 个缺口均为单帧缺口，对应接收间隔约 350～419 µs，缺口比例约
0.0003%。Publisher 没有跳周期，因此缺口发生在 Writer 之后至应用 Reader
可见之前。BestEffort + KeepLast(1) 下，DDS 的 `sample_lost=0` 不代表
应用序号必然连续，应以 `sequence_gaps` 为准。

5 kHz 已越过本测试的“零序号缺口”边界，但这不影响正式 1 kHz 结论。
已有结果表明 1 kHz 长测通过、2 kHz 三轮通过，5 kHz 开始出现少量缺口。

### 7.4 ddsperf 1 kHz 消息大小压力

网卡 `enp0s31f6` 为 1000 Mb/s 全双工、MTU 1500。固定频率 1 kHz，
逐步增加样本大小。表中的 P99 是每秒窗口 P99 的平均值。

| 大小 | 单向有效载荷 | RTT/2 平均 | RTT/2 P99 | RTT/2 最大 | 回复速率 | 周期超时计数 | 传输计数 | 单向 1 ms 筛选 |
|---:|---:|---:|---:|---:|---:|---:|---|---|
| 32 KiB | 262.144 Mb/s | 515.383 µs | 590.356 µs | 952.455 µs | 998.000/s | 42,818 | 全部为 0 | 通过 |
| 48 KiB | 393.216 Mb/s | 671.827 µs | 740.038 µs | 851.886 µs | 998.067/s | 59,186 | 全部为 0 | 通过 |
| 56 KiB | 458.752 Mb/s | 716.486 µs | 807.508 µs | 898.980 µs | 997.967/s | 59,182 | 全部为 0 | 通过 |
| 60 KiB | 491.520 Mb/s | 736.148 µs | 809.531 µs | 993.506 µs | 997.967/s | 59,183 | 全部为 0 | 临界通过 |
| 62 KiB | 507.904 Mb/s | 732.539 µs | 792.662 µs | 986.297 µs | 997.967/s | 59,184 | 全部为 0 | 临界通过 |
| 64 KiB | 524.288 Mb/s | 819.955 µs | 966.007 µs | 1112.443 µs | 997.950/s | 59,184 | 全部为 0 | 未通过 |

“传输计数全部为 0”表示 `discarded/rexmit/Tthrottle/Nthrottle` 均为 0。
RTT/2 只是对称链路条件下的单向时延近似，不是经过 PTP 同步得到的真实
单向时延。

ddsperf 在固定 1 kHz 模式下，如果上一个 Pong 未在下一个 Ping 周期前返回，
就增加 `ping timed out`，即使该 Pong 稍后仍然到达。因此大消息测试中的
周期超时计数不能解释成丢包数，也不能使用
`timeouts / (RTT samples + timeouts)` 计算丢包率。大消息测试仍收到约
998 个 RTT 样本/秒，且传输计数为 0。

60 秒短测显示 60～62 KiB 已非常接近 1 ms 边界，单次最大值又受调度随机
峰值影响，不能作为稳定工程上限。综合结果选择 56 KiB 作为高负载余量
候选，48 KiB 作为更保守的长期候选：

```text
极限区间          60～64 KiB
高负载余量候选    56 KiB
保守长期候选      48 KiB
```

下一步计划是执行 `1 kHz × 56 KiB × 1800 秒` 连续测试。验收条件为
RTT/2 最大值小于 1 ms、回复速率稳定、所有传输计数为 0。该 30 分钟测试
尚未提供结果，因此不计入本报告已完成项目。
